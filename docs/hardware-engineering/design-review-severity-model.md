# 设计评审严重度模型

## 适用场景

适用于原理图 review、layout review、bring-up 问题分级和维护团队协作。

## 关键规则

- 严重度应反映 `失效后果` 和 `修复窗口`，而不是评论者情绪。
- 最少应区分 `阻塞量产/上电`、`高风险缺陷`、`建议优化` 三档。
- 评审项必须可执行，不能只有“建议检查一下”。
- 一个好的分级模型能减少争论，把讨论拉回工程事实。

## 推荐分级

- `ERROR`: 会导致不上电、损坏器件、严重不兼容、协议必然失败或无法量产。
- `WARNING`: 有较大风险、边界条件不稳、可靠性不足或调试成本高。
- `NOTE`: 优化建议、文档改进、可维护性提升、未来变型友好性。

## 评审清单

- 每个问题是否写明了影响、证据和建议动作。
- 是否区分“理论更优”和“必须修改”。
- 问题是否绑定到具体页、网络、器件或接口。
- 是否明确修复前是否允许继续 layout、打样或 bring-up。

## 常见失误

- 把所有问题都标成最高严重度，结果没人再相信分级。
- 只给结论不给证据，团队无法复核。
- 评论停留在偏好层面，没有连接到失效模式。
