# Extractor Domain Map

Last audited from code: 2026-05-17.

This document maps extractor modules to ownership and current implementation
status. It is about `extractors/`, not public export coverage. Public coverage
is listed in [Current State](current-state.md).

## Registry Order

`extractors/__init__.py` registers extractors in this order:

1. `ElectricalExtractor`
2. `PinExtractor`
3. `ThermalExtractor`
4. `DesignContextExtractor`
5. `RegisterExtractor`
6. `TimingExtractor`
7. `PowerSequenceExtractor`
8. `ParametricExtractor`
9. `ProtocolExtractor`
10. `PackageExtractor`
11. `DesignGuideExtractor`

Order matters because derived domains may depend on earlier results.

## Domain Status

## Ownership Rules

### `electrical`

Owns:

- component identity when extracted from the datasheet
- absolute maximum ratings
- electrical characteristics
- numerical DRC hints derived from electrical tables

### `pin`

Owns:

- normal-IC logical pins
- package-to-pin maps for normal ICs
- pin direction, signal type, unused treatment

For FPGA package pinout truth, use `data/extracted_v2/fpga/pinout/` generated by
vendor parsers, not `PinExtractor`.

### `thermal`

Owns:

- thermal resistance style metrics
- power dissipation entries
- thermal normalization derived from electrical/absolute-maximum data

### `design_context`

Owns:

- application page candidates
- external component hints
- layout hints
- equation hints
- topology hints

### `design_guide`

Owns:

- standalone vendor design-guide rules
- rail merge guidance
- power sequencing rules
- clock/configuration/layout design rules

### `timing`

Owns:

- setup/hold/propagation delay
- switching windows
- clock timing constraints

### `power_sequence`

Owns:

- rail ordering
- startup/shutdown constraints
- UVLO, enable, soft-start facts where explicitly modeled

### `protocol`

Owns:

- interface types
- bus/protocol capabilities
- signal-level interface semantics

### `parametric`

Owns:

- comparison-oriented normalized specs
- selector-friendly summaries

Current caveat: it is not effectively wired into `pipeline_v2.py`.

### `package`

Owns:

- package metadata beyond raw pin records
- mechanical/package facts
- land pattern and marking style data when extracted

### `register`

Owns:

- register maps
- bitfields
- access/reset metadata

## Boundary Rule

If a fact is package-specific FPGA pinout truth, it belongs in a vendor parser
output.

If a fact is a generic numerical datasheet parameter, it usually starts in
`electrical` and may be normalized into `thermal`, `parametric`, or DRC hints.

If a fact affects board integration behavior, it belongs in `design_context`,
`design_guide`, `protocol`, or `power_sequence`.
