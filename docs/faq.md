# OpenDatasheet FAQ

## Where should I start?

Read these in order:

1. `docs/current-state.md`
2. `README.md`
3. `GUIDE.md`
4. `docs/architecture.md`
5. `docs/index.md`

If you consume public exports, go next to
`docs/sch-review-integration.md` and `schemas/sch-review-device.schema.json`.

## Which docs are authoritative?

For repository state, counts, architecture, and data provenance, use:

- `docs/current-state.md`
- `docs/architecture.md`
- `docs/sch-review-integration.md`
- `schemas/sch-review-device.schema.json`

Some older docs are retained as design history or case notes. They may contain
old counts or old pipeline assumptions.

## What is the current schema target?

Current checked-in public exports use `device-knowledge/2.0`.

The validator still accepts:

- `sch-review-device/1.0`
- `sch-review-device/1.1`
- `device-knowledge/2.0`

That compatibility is for old payloads and downstream migration. New public
exports should be generated as `device-knowledge/2.0`.

## How do I validate the local checkout?

For export validation:

```bash
python scripts/validate_exports.py --summary
```

For the broader local gate:

```bash
./scripts/run_checks.sh
```

For environment checks:

```bash
python scripts/doctor.py --dev
```

## Do I need `GEMINI_API_KEY`?

Only for model-backed extraction flows such as:

```bash
python pipeline_v2.py <pdf-path>
```

You do not need a Gemini key to validate exports, read public exports, parse
FPGA pinout files, or regenerate public exports from already checked-in
intermediate JSON.

## Is the repository a pure Gemini Vision pipeline?

No.

The repository currently has multiple ingestion paths:

- historical model-labeled intermediate JSON
- explicit manual profiles
- deterministic FPGA pinout parser outputs
- optional model-backed extraction code
- export normalization code

Do not describe all current data as Gemini Vision output.

## When should I regenerate `data/sch_review_export/`?

Regenerate when you change:

- `scripts/export_for_sch_review.py`
- `scripts/normal_ic_contract.py`
- export normalization helpers
- schema semantics
- checked-in extracted inputs used by the exporter
- FPGA pinout parser outputs

Typical commands:

```bash
python scripts/export_for_sch_review.py
python scripts/validate_exports.py --summary
```

Usually do not regenerate for documentation-only changes.

## What if pytest fails before running repository tests?

On Windows, a local Python installation may fail while auto-loading unrelated
third-party pytest plugins, for example because `_sqlite3` is missing. For a
focused repository test run, use:

```powershell
$env:PYTEST_DISABLE_PLUGIN_AUTOLOAD='1'
python -m pytest -q
```

That workaround does not hide repository test failures; it prevents external
pytest plugins from breaking collection before tests start.

## Where do FPGA facts come from?

FPGA package pinout facts come primarily from deterministic vendor parsers and
checked-in normalized pinout JSON under:

```text
data/extracted_v2/fpga/pinout/
```

The public FPGA exports are generated by:

```bash
python scripts/export_for_sch_review.py
```

Use:

## When should a change become an architecture discussion?

Escalate before proceeding if you plan to:

- change schema semantics
- remove compatibility with old schema versions
- change the meaning of `normal_ic` or `fpga`
- change public export field semantics
- require downstream consumers to change code
- perform a large semantic regeneration
