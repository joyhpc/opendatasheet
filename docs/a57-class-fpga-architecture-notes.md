# A57-Class FPGA Architecture Notes

> 从 `joyhpc/A57-docs` 这类车载相机老化 / 域控测试平台的真实需求出发，讨论 FPGA 选型、分层和板级约束。本文不是营销对比，也不是 RTL 视角，而是站在架构工程师 / 原理图 owner 角度，回答这类系统到底在消耗什么 FPGA 资源。

## 为什么 A57 这类项目不是普通 FPGA 选型题

如果只看器件页，A57 看起来像一个“多协议 + 多板卡”的复杂平台：

- 老化侧需要覆盖 `MIPI D-PHY`
- 域控侧需要覆盖 `eDP / HDMI / OpenLDI / CAN / Ethernet / UART`
- 未来还要兼容更多车载 `SerDes` 模组
- 解码板是可插拔 `x16 PCIe AIC` 形态
- 系统目标有 `16 / 32 / 64` 路摄像头扩展
- 32 路和 64 路版本还要带 `4 x 10G` 光输出

但从架构上看，这类项目的真正难点不是“功能很多”，而是下面四件事同时成立：

1. 输入侧和输出侧都在高速化，而且协议并不统一。
2. 通道规模是平台级的，不是单板 demo 级的。
3. 各板之间必须解耦，不能让某一个 FPGA 把全系统绑死。
4. 设计目标里已经隐含了未来 3 到 5 年的接口演进。

所以 A57 这种系统不是在找“一颗最强 FPGA”，而是在找一个**分层异构 FPGA 体系**。

## A57 类系统真正消耗的不是 LUT，而是四种“体系资源”

### 1. MIPI 资源不是 lane 数，而是可落板的 D-PHY 资源

很多项目把 MIPI 资源理解成“有几个 RX lane”。这在 A57 类项目里明显不够。

这里真正消耗的是：

- 能否把 D-PHY 相关引脚落在合适 bank
- 同一个器件上，MIPI 和 DDR、普通 GPIO、控制接口是否会互相挤占
- 高速差分和时钟在封装逃逸后，走线长度、skew、参考平面是否还能维持余量
- 多路虚拟通道或并发突发场景下，核心电源和 IO 电源是否还能稳住

A57 可行性评估里给出的一个很关键的判断是：`AU15P` 在 `160 mil` 条件下连续跑 `MIPI D-PHY 3.5 Gbps` 两天无错，因此推断 `200 mil` 下跑 `2.5 Gbps/lane` 具备现实可行性。这个结论很有代表性，因为它说明团队已经不是停留在“器件支持 MIPI”这一层，而是在用板级链路余量去判断能否量产。

这类系统里，MIPI 的核心问题永远不是“支不支持”，而是：

> 你是不是还能给 DDR、控制面、复位、调试和未来扩展留下足够的 bank 与布线余量。

### 2. DDR 不是容量附件，而是流量缓冲器和故障取证池

A57 文档里一个非常值得保留的判断是：每个 MIPI 通道希望保证 `0.25 GB DDR` 缓冲，并给出 `2 x 1 GB, 16-bit DDR` 这一类容量级别的配置思路。

这说明 DDR 在这里不是简单的 frame buffer，而是同时承担四个系统职责：

- 吸收输入突发和输出不均衡造成的瞬时带宽冲击
- 作为多帧缓存，保证分析任务和转发任务可以解耦
- 保存坏帧、缺帧、异常帧上下文，支持本地诊断和回放
- 在路由板 / 汇聚板场景里，给未来 ingress 大于 egress 的演进预留弹性

这和很多消费类 FPGA 板很不一样。后者常常把 DDR 当成“有更好，没有也能做”。A57 这种系统里，DDR 是架构弹性本身。

更直接地说：

> 如果没有足够可控的外部 DDR，A57 类系统的很多“本地分析、坏帧定位、通道去耦、后端限流”能力都会退化成纸面能力。

### 3. SerDes 资源不是速率 headline，而是全链路余量所有权

A57 对 `eDP 16 Gbps/lane`、`4 x 10G` 光输出、以及多种车载高速接口兼容的要求，决定了 SerDes 在这个项目里不是辅助能力，而是系统骨架的一部分。

这类系统里，SerDes 资源至少包含：

- 你拿到的是哪组 GT / H-Tile / Q0 lane
- 对应 refclk 是谁拥有，谁生成，谁分发
- 封装逃逸后能不能进连接器、背板、开关、redriver、光模块
- 板上插损、反射、串扰和 switch/mux 引入的 margin 损失
- 首板调试时有没有独立链路可观测性

A57 的可行性评估并没有把 eDP 风险理解成“GT 速率够不够”，而是明确落到链路损耗、开关切换、skew、接收余量这些板级问题上。这个判断是对的。

GWBRGIC 的审核材料更把这件事讲透了：`GW5AT-LV60UG225` 上 `SERDES Q0 (Bank H)` 对外走四路收发，`OSC5 200 MHz LVDS` 提供 SerDes refclk，同时还出现了 `OSC4` 频率标注歧义、`J4` 默认 `NC` 影响回路调试、若 refclk 装错则链路可能直接不锁的问题。

这正是 A57 类项目里最容易被忽略的事实：

> SerDes 风险不在 datasheet 首页，而在 refclk、连接器、装配选项和测试闭环是否真正归属于架构设计。

### 4. MCU-FPGA 边界不是外设连接，而是系统复杂度分配

`STM32H745IIK6 -> XCKU3P` 的 `FMC` 资源分配报告非常典型。它刻意把 MCU 端简化为一个精简异步 SRAM 风格接口，省掉 `FMC_CLK / FMC_NWAIT / FMC_NBL0 / FMC_NBL1` 等信号，把协议处理、时序吸收和 pin-swapping 灵活性都压到 FPGA 一侧。

这意味着架构上的一个明确取舍：

- MCU 负责控制与系统管理
- FPGA 负责吸收接口复杂度和板级布局自由度

这不是为了“省几根线”这么简单，而是因为在多板、多协议、多通道系统里，**把复杂性收敛到 FPGA** 比把复杂性摊到 MCU 和 PCB 上更可控。

对架构工程师来说，这一条特别重要：

> A57 类系统的 MCU-FPGA 边界应当故意做成不对称结构，尽量让 FPGA 成为协议和引脚灵活性的缓冲层。

## A57 实际上天然要求异构 FPGA 分层

从资料里的器件角色来看，A57 并没有走“单一 FPGA 家族统一全系统”的路线，而是自然演化成至少三层：

### 第一层：中心汇聚 / 主处理 FPGA

代表器件是 `XCKU3P-2FFVB676E`。

它承担的通常不是单一接口适配，而是：

- 多通道视频汇聚
- 更重的数据路径和本地分析
- 与 DDR 的主耦合
- 与光口 / 高速输出 / 上层主机的主耦合
- 系统级时钟、复位、下载和调试协调

这类位置适合放在资源、时序余量和工具成熟度都更强的家族上。原因不是“高端 FPGA 比较强”这么空泛，而是因为中心层一旦失误，会把系统的 DDR、SerDes、功耗、热和 bring-up 全部绑在一起。

### 第二层：路由 / 域控显示 / 协议重整 FPGA

代表器件是 `XCAU15P-SBVB484-2-E` 这类器件。

它更像是系统里的高速路由节点和接口重整节点，重点通常是：

- eDP / HDMI / OLDI 等显示类高速接口
- 板间协议分配和切换
- 某些高速输入的前级整理
- 为中心 FPGA 分担封装和引脚压力

这个层级未必承担最大算法负载，但会直接决定系统的银行规划、差分对规划和热设计压力。

### 第三层：边缘桥接 / 解码板 / 协议 Shim FPGA

代表器件来自 `Gowin`、`Lattice` 这类更适合边缘桥接的家族，例如资料中的 `GW5AT-LV60UG225`、`LFE5U-25`、`DTU525` 一类角色。

这层 FPGA 靠近解码器、相机模组或某个特定输入接口，职责更偏向：

- 协议适配
- lane 重映射
- C-PHY / D-PHY / SerDes 的局部桥接
- 特定接口的首段电气收敛
- 将复杂性隔离在小板 / 子板，而不是带回中心主板

这个分层对 A57 很关键。因为它说明：

> 对这类平台来说，最优解往往不是“全系统同一家 FPGA”，而是“让最贵、最重、最难改的职责集中在最稳的器件，把协议碎片化问题留在边缘层处理”。

## 从 A57 视角看四家 FPGA，不是强弱关系，而是岗位分配

### AMD：适合中心汇聚层，也适合高速路由层

从 A57 资料看，`KU3P` 和 `AU15P` 分工非常清楚。前者更像中心主处理 / 汇聚内核，后者更像高速路由与显示分发节点。

AMD 在这类项目中的价值，主要不是 LUT 多，而是：

- 高速接口和主干数据路径可放在较少器件上闭合
- 中心层和路由层都能维持较强的时序与工具可预测性
- 当系统规模提升到 `32 / 64` 路时，更容易维持架构连续性

但代价也很明确：

- bank 规划必须前置
- DDR 与高速 IO 的资源竞争必须前置
- 包封、散热、供电和 refclk 拥有权必须前置

AMD 适合 A57 里“不能掉链子”的中心层，不适合用来滥做所有边缘小功能。

### Intel / Altera：更适合做平台节点，而不是被动桥接件

如果把 A57 类需求映射到 Intel / Altera，比较合适的位置通常不是最边缘的轻桥接，而是那种同时需要：

- 软件控制面
- 外部 DDR
- 高速 IO
- 平台节点自治

的板位。

这也是为什么 Intel / Altera 在这类平台里更应该被理解为“平台组件”，而不是“单纯逻辑替代品”。如果系统希望把控制面、数据面、存储面更紧地捏在一起，或者希望减少独立 CPU / MCU 数量，它会很有吸引力。

但使用前提也更严格：

- family 级资料不能直接替代 package 级事实
- device capability 不能直接替代 ordering variant 事实
- HPS / EMIF / transceiver 的存在与否必须按具体料号核实

也就是说，Intel / Altera 不是不能用于 A57 类项目，而是更适合那些从一开始就接受“平台节点化”设计的板卡。

### Lattice：适合边缘适配、轻量桥接、接口聚焦型板卡

如果需求是：

- 摄像头侧或显示侧的局部协议适配
- 不想把板级复杂度拉到主板
- 更强调功耗、成本、外围简化

那 Lattice 往往适合放在 A57 体系的边缘层。

它的优势不是“能取代中心 FPGA”，而是能让边缘职责保持边缘职责，不把系统重新中心化。

这对解码板、小型桥接板、协议收敛板很有价值。

### Gowin：适合成本敏感或国产链路约束下的边缘桥接层

GWBRGIC 的案例很说明问题。`GW5AT-LV60UG225` 在板上承担的是“靠近解码器、收 MIPI、出 SerDes”的桥接角色，而不是全系统中心。

这个位置很合理，因为 Gowin 在这类系统里的价值通常来自：

- 成本或供应链约束下的可得性
- 适合作为特定接口的小板桥接器件
- 可以把某一段协议复杂性先就地收敛

但它的工程代价也必须老实承认：

- refclk、上电时序、装配选项、NC 预留的风险需要更严格审核
- 文档和实板一致性需要更仔细复核
- 首板 bring-up 需要更强的观测和回路验证设计

所以 Gowin 并不是不能做高速，而是更适合被放在“局部收敛、局部验证、局部隔离”的位置上。

## A57 给架构评审的真正启发

### 启发一：不要把 family summary 当成 package 许可

A57 这种平台的最大风险之一，就是把“家族支持某协议”误用成“手上这个封装、这个料号、这个板层一定能做成”。

这在下面几类问题里最容易出事：

- 以为 MIPI lane 够，结果 D-PHY 适配 bank 不够
- 以为 GT 速率够，结果 refclk 所在组和连接器逃逸不匹配
- 以为 DDR 能接，结果中心层已经被别的功能占掉关键 bank
- 以为边缘 FPGA 能替代中心 FPGA，结果本地分析和缓存能力失衡

### 启发二：DDR、MIPI、SerDes 必须一起审，不要分三个章节各自乐观

A57 的真实系统压力来自这三者耦合：

- MIPI 输入制造突发
- DDR 吸收突发并支撑分析
- SerDes / eDP / 光口把结果送出去

如果评审时把它们拆开看，结论很容易都“单项可行”；但一旦放回一块板上，功耗、bank、时钟、热和走线就会彼此打架。

真正有效的评审方式是把它们作为一个三角共同体审查：

- MIPI 输入峰值时，DDR 写入有多少安全余量
- DDR 忙的时候，输出路径有没有足够回压 / 限流策略
- SerDes / eDP 使用的 refclk、封装区块、连接器位置是否挤压了 MIPI 和 DDR 的可实现性

### 启发三：A57 类项目应该主动采用“中心稳、边缘活”的分层策略

这类平台最怕两种错误：

- 把所有职责压到一颗中心 FPGA，导致封装、供电、热和调试一次性爆炸
- 为了追求统一家族，把边缘板也做成和中心板一样重

更合理的策略是：

- 中心层选稳定、余量足、可持续扩展的家族
- 边缘层优先解决协议收敛和成本问题
- 路由 / 显示 / 光口层单独看高速拓扑，不与边缘桥接混成一个层级

这其实就是 A57 资料里已经体现出的思路，只是原始文档是分散在不同评估和审核报告里。

### 启发四：MCU 不是在和 FPGA 平分复杂度，而是在替 FPGA 守住系统边界

H745 到 XCKU3P 的 FMC 设计给出的启示很直接：控制器不应该承担高速系统的细枝末节时序压力，它应该承担：

- 上下电与系统状态管理
- 配置、异常、升级和维护入口
- 简化的软件控制接口

而 FPGA 负责：

- 吃掉协议差异
- 吃掉 pin-swapping 和板级映射复杂性
- 吃掉跨板接口的时序不整齐性

这能显著减少“控制器选型牵动整板布线”的耦合。

## 给 A57 类项目的架构评审清单

如果以后继续做这类平台，我建议在原理图冻结前至少逐项确认下面这些问题：

1. 对每一类高速接口，是否已经区分了 family capability、device boundary、ordering feature、package pin fact 四层事实。
2. 对每一块板，是否明确了它属于中心汇聚层、路由显示层，还是边缘桥接层。
3. 每个 MIPI / C-PHY / D-PHY 接口是否已经和具体 bank、具体封装逃逸、具体时钟源绑定。
4. 每个 SerDes 链路是否已经明确 refclk 拥有者、连接器路径、redriver / switch 路径和首板测试闭环。
5. DDR 是否按系统弹性资源来预算，而不是按静态容量来预算。
6. 是否给坏帧、缺帧、本地分析、回放和调试日志留出了真实缓存余量。
7. MCU-FPGA 边界是否刻意做了简化，避免 MCU 侧背上过多时序和引脚负担。
8. 所有 `NC` 选项、预留电阻、时钟替代位、调试连接器是否都已经被纳入 bring-up 策略，而不是停留在“先空着”。

## 结论

A57 这类项目最值得保留的工程经验，不是“哪家 FPGA 更强”，而是下面这句更硬的判断：

> 对多通道车载相机老化 / 域控测试平台来说，FPGA 不是单芯片选型问题，而是一个分层异构系统设计问题。

在这个问题里：

- `DDR` 是系统弹性资源，不是附属容量
- `MIPI` 是 bank 与布线余量问题，不只是 lane 数
- `SerDes` 是 refclk 与链路所有权问题，不只是 headline Gbps
- `MCU-FPGA` 边界是系统复杂度分配问题，不只是一条控制总线

因此，对 A57 类平台最合理的 FPGA 策略通常是：

- 中心层优先保证稳定性、扩展性和高速主干余量
- 路由 / 显示层优先保证高速拓扑可实现
- 边缘层优先解决协议碎片化和成本
- 不强求全系统单一家族，而强调职责分层清晰

这也是为什么从 A57 的资料回看四家 FPGA 时，真正重要的不是参数表胜负，而是它们分别更适合站在系统的哪一层。
